# 課務 CRM 系統 — 產品需求文件（PRD）
> **撰寫：Agent A（產品）** ｜ 版本 v1.0 ｜ 對應原型：`出缺勤系統.html`
> 本文件依專案唯一事實來源 `CLAUDE.md` 之實作現況撰寫；與程式衝突時以 `CLAUDE.md` 為準。

## 1. 背景與問題

HBR 學院以「個案共學會、領導者俱樂部、實戰班」等系列課程為核心，過往課務、招生、客戶與成效資料分散於 Excel 與多個後台，造成：
- 課程／期別、學員名單、營收分散，無法即時掌握各期狀態。
- 招生對象（潛客）與既有客戶分流不清，重複接觸或漏接。
- 各期招生成效（說明會、報名、成交）無系統化追蹤，無法跨期比較。
- 帳號與權限無控管，敏感資料（成本／損益）人人可見。

本系統將上述整合為單一前端 CRM 原型，作為正式開發（PHP 後端）的需求與互動藍本。

## 2. 產品目標

| 目標 | 衡量方式 |
|------|----------|
| 單一介面管理課程／期別與學員 | 課程列表、期別狀態、學員人數即時呈現 |
| 招生對象分流與轉換可視 | 潛客→顧客轉換規則明確，期別歷程可追溯 |
| 各期招生成效可量化、可比較 | 報名追蹤＋說明會成效（出席率／轉換率）＋本期彙總 |
| 權限控管 | 登入閘門＋角色（superuser／staff），帳號管理僅 superuser |

## 3. 範圍

### 3.1 本期範圍（已實作）
- 登入 / 帳號權限管理
- 課程管理（列表／篩選／詳情／建課精靈／訂單匯入學員）
- 期數成效管理（商品→期別→報名追蹤／說明會成效，三層）
- 潛客管理（來源／狀態／匯入／批次／匯出）
- 客戶管理（類型／備註標籤／購買課程／匯入／批次／匯出）

### 3.2 非範圍（Non-Goals，本期不做）
- 出缺勤登記／點名（已移除，本階段無需求）
- 真實後端 API、密碼雜湊、金流串接
- 手動逐筆新增/刪除學員（學員一律由後台訂單匯入）
- 潛客手動「轉為顧客」按鈕（僅由訂單匯入自動轉換）

## 4. 使用者與角色

| 角色 | 說明 | 權限邊界 |
|------|------|----------|
| 系統管理員 superuser | 全權；唯一可進「帳號權限管理」 | 可新增/編輯/停啟/重設密碼/刪除帳號 |
| 課務人員 staff | 日常操作課程／成效／潛客／客戶 | 看不到帳號權限管理（側欄不顯示） |

**安全規則**：至少保留一個啟用中的 superuser（不可停用/降權/刪除最後一個）；不可停用/刪除自己；新建帳號預設 staff。

## 5. 領域模型（核心名詞）

| 名詞 | 定義 | 關鍵欄位 |
|------|------|----------|
| 課程／期 Course | 一檔課程的一期 | `id`(代碼)、`name`、`type`(類別)、招生期間 `recruitStart/End`、開課期間 `startDate/endDate`、`orders`、`revenue`、`costs`、`sessionsList`、`seminarCount`、`recruitSources` |
| 場次／單堂 Session | 一期下一堂課 | `no`、`date`、`time`、`cases[]`、`done`、`cancelled` |
| 個案 Case | 場次內個案 | `title`、`subtitle`、`facilitator`(講師)、`org` |
| 訂單 Order | 一位學員＝一筆訂單 | `name`、`company`、`amount`、`src` |
| 客戶 Customer | 已購買者 | `id`(H+流水)、`type`(學員/VIP)、`purchased[]`、`note`、`noteTags` |
| 潛客 Prospect | 未成交開發對象 | `id`(P+7位)、`sources[]`(來源)、`status`(狀態)、`note` |
| 期成效 Perf | 一期的招生追蹤＋說明會成效 | `poolMembers[]`、`track{}`、`seminars[]`、`newProspectIds[]` |
| 帳號 User | 系統使用者 | `email`、`password`、`role`、`active` |

## 6. 功能需求摘要（詳見《功能細項表》）

1. **登入**：Email＋密碼＋記住我；mock 驗證；demo 帳號提示。
2. **帳號權限管理**（僅 superuser）：CRUD＋停啟＋重設密碼＋安全規則。
3. **課程管理**：狀態/類別/年度篩選＋搜尋；兩步驟建課精靈（招生＋開課兩段日期、自動判狀態）；課程詳情（KPI、場次表、停課、損益表、訂單匯入學員）。
4. **期數成效管理**：三層；①報名追蹤（匯入驅動、身分動態判定、當期備註、說明會欄）；②說明會成效（每場獨立計出席率/轉換率＋本期彙總，僅個案共學會）。
5. **潛客管理**：狀態分頁＋來源篩選＋搜尋＋排序；列上就地改狀態/來源；匯入/批次/匯出。
6. **客戶管理**：類型分頁＋搜尋＋排序；備註標籤；購買課程依類別分頁；期別歷程時間軸；匯入/批次/匯出。

## 7. 狀態與業務規則

- **課程狀態**（自動）：今天 < 開課日＝招生中；開課期間＝上課中；結課後＝已結案；**停課**為手動旗標、優先於日期。延長改於編輯課程修改日期。
- **營收**＝該期訂單金額加總，不可手動輸入。
- **學員來源**：唯一入口為課程詳情「新增學員（後台訂單匯入）」。
- **潛客→顧客轉換**：訂單匯入時比對潛客（原型以姓名＋公司；正式版改訂單編號＋帳號），比中者轉為顧客、purchased 加本期課名、本期池標記已成交。
- **身分動態判定**（報名追蹤）：依 person×期 計算「舊客未報名／舊潛客／新潛客」。

## 8. 資料持久化（原型）

前端 localStorage：`hbr_courses_v1`、`hbr_customers_v2`、`hbr_prospects_v1`、`hbr_perf_v2`、`hbr_users_v1`、`hbr_session_v1`、`hbrCRM.suspendedCourses`。正式版以 API/DB 取代（見根目錄 `API契約.md`）。

## 9. 階段規劃

- **P0（現狀，原型）**：上述五模組 UI/UX 與前端資料邏輯。
- **P1（後端串接）**：替換 `app/data.js` 各函式為 fetch；同步→非同步改寫；訂單編號＋帳號去重與轉換；密碼雜湊與權限驗證。
- **P2（延伸）**：出缺勤／點名（若有需求再開）、報表匯出強化、通知。

## 10. 風險與待確認

- 編號規則（會員/訂單編號位數、前綴、流水）待需求方拍板。
- 潛客→顧客比對在正式版須有訂單編號＋帳號，否則同名同公司可能誤判。
- localStorage 為單機資料，多人協作須待後端。
