## 0. 協作原則（務必遵守）

1. **逐步記錄**：每次規劃／功能變動，都要更新本檔對應章節（用可驗證的 given / when / then 敘述，不用模糊形容詞）。
2. **不要幻想（no hallucination）**：只記錄與實作「確定存在」的行為。需求不明、或與現況矛盾時，**先向 user 提問確認**，不要自行臆測補完。
3. **出錯就提出**：發現資料結構、需求或既有程式有衝突／錯誤時，停下來向 user 確認，不要硬做。
4. **完成後驗證**：寫完規格或做完功能，另開一個 verifier agent 確認「規劃與實作是否一致、是否正確」。
5. **這是原型（prototype）**：目前**沒有串接任何後端 API**，資料皆為前端 mock；標示「示範」者為假資料或假流程。正式開發時依「資料契約／後端對應」章節落地。

### 0.1 UI/UX 與效能優化原則（日後修改一律比照）

> 任何修改都要兼顧「體驗一致」與「主機負擔最小」，且**不得自行新增或刪減功能**（除非 user 明確要求）。

**UI／UX**
- **沿用既有視覺語彙**：一律重用 `app/shell.jsx` 的共用元件與 `app/styles/tokens.css` 的 `var(--*)` token；**勿自創顏色、字體、陰影、圓角**。新樣式先找既有 token／class。
- **跨模組一致**：同類互動用同一套樣式（如篩選＝Topbar tabs＋工具列下拉、列表＝`.tbl`、彈窗＝`Modal`、空狀態＝置中圖示＋說明＋清除鈕）。
- **保留回饋狀態**：互動元件需有 hover／`:active`／`:focus-visible`（鍵盤）狀態；破壞性動作用 `--danger-*`。
- **保留輔助狀態**：搜尋/篩選要保有「N / M」計數、清除鈕、空結果提示，勿移除。
- **動效節制**：過場短（≤200ms）、無裝飾性無限迴圈；一律受 `@media (prefers-reduced-motion: reduce)` 保護。

**效能（避免吃主機）**
- **資料讀取走 `app/data.js` 並用快取**：`loadCourses/loadCustomers` 已做記憶體快取，寫入（`save*`）時同步更新、失效；**勿在 render 中重複 `JSON.parse(localStorage)`**。熱路徑（點名登記）尤其注意。
- **不可就地 mutate 快取回傳值**：一律用 `map/filter/spread` 等不可變操作，避免污染共用參照。
- **減少重算與重繪**：跨 render 的衍生計算用 `React.useMemo`（依賴鍵明確）；避免在大列表每列做昂貴運算；DOM 數量保持精簡。
- **避免不必要的全域重渲染**：狀態盡量就近，避免一次 setState 觸發整頁重算。

**流程**
- 改完依 §0.4 驗證（載入無錯誤、流程正常、無視覺破壞），並更新本檔對應章節與變更紀錄。

## 1. 技術架構

- 單一 HTML（`出缺勤系統.html`）以 React 18 + Babel standalone（`type="text/babel"`）載入多個 `app/*.jsx`。
- **各 babel script 各自獨立作用域**：跨檔共用的元件／函式一律掛到 `window`（檔尾 `Object.assign(window, {...})` 或 `window.X = ...`），並以裸名引用。新增檔案務必加 `<script>` 並注意載入順序（被依賴者先載）。
- 共用 UI 元件在 `app/shell.jsx`（`Topbar / ToolBar / SearchBox / KPI / Avatar / StatusChip / Modal / Field / Icon` 等）。
- 樣式 token 在 `app/styles/*.css`，用 `var(--*)`；勿自創顏色。

## 2. 領域模型

| 名詞 | 說明 |
|------|------|
| 期／課程 Course | 一檔課程（如「個案共學會・第16期」）。鍵 `id`＝課程訂單編號。 |
| 場次／單堂 Session | 一期下的一堂課，有日期、時間。`sessionsList[]`（新）或 `SESSIONS[courseId]`（種子）。 |
| 個案 Case | 一個場次內的個案：`{title 主題, subtitle 情境副標, facilitator 講師, org 服務單位}`。每場可多個。`facilitator` 為資料鍵名（沿用），UI 一律顯示「講師」。 |
| 上課名單 Roster | 該期學員清單。**課程層級**，套用所有場次。 |
| 訂單 Order | 一位學員＝一筆訂單；金額加總＝課程營收。 |
| 客戶 Customer | CRM 主體；`id`＝會員編號（H + 流水號）。類型＝學員／VIP（潛客已分流至潛客管理）。 |
| 潛客 Prospect | 尚未成交的開發對象；獨立模組與 store。`id`＝潛客編號（P + 7 位流水）。無購買課程，帶**來源**（標籤，可多選）與**狀態**（單選開發階段）。
| 師資 Faculty | 講師主檔 `FACULTY[{id,name,org}]`；課程個案的講師欄可從此選用。 |

## 3. 課程管理 規格

### 3.1 課程列表
- 欄位（依序）：**課程訂單編號／課程名稱／學員人數／課程起訖日／狀態**，最右為箭頭 icon（進詳情）＋紅色刪除 icon。
- 點整列或箭頭 → 進入課程詳情頁。
- 工具列：搜尋（名稱/代碼）、共 N 門課程；右上「匯入課程（Excel）」（示範流程）、「新增課程」。
- **狀態篩選分頁**（Topbar tabs，仿客戶管理）：全部／招生中／準備開課／上課中／已結案／停課，各帶數量與狀態色點；狀態由 `deriveCourseStatus` 即時推導。
- **類別與年度篩選**（工具列下拉）：類別（全部類別／個案共學會／實戰班／領導者俱樂部，讀 `COURSE_CATEGORIES`，舊值經 `categoryOf` 映射）、年度（全部年度／2022～「今年與課程最大開始年份」，動態推導；跨年或新增未來年度課程會自動出現）。
- 狀態、類別、年度、搜尋可並用；有篩選時計數顯示「N / M 門課程」並出現「清除篩選」，空結果有空狀態提示。
- **右鍵任一列**選單：**編輯課程／複製課程／刪除課程**。

### 3.2 建立／複製／編輯課程（兩步驟精靈，`NewCourseModal`，`mode: new|copy|edit`）
步驟：**① 基本資訊 → ② 課程場次**（本階段不設成本）
> 設計意圖：課程剛建立可能尚未對外販售，**無學員為正常狀態**；學員改由課程建立後於課程管理以「新增學員（後台訂單匯入）」載入（見 3.3a）。精靈內不再編輯學員；**本階段不設成本**（移除「成本結構」步驟）。
- **① 基本資訊**：課程代碼*、課程名稱*、**類別***（個案共學會／實戰班／領導者俱樂部，`COURSE_CATEGORIES`）、開始日期*、結束日期*。
  - 必填阻擋：按「下一步」若未完成顯示紅框＋訊息並擋下；結束需 ≥ 開始；代碼重複即時提示（編輯模式排除自己）。
  - **類別儲存於 `type` 欄**；舊值（個案共學/會員制）由 `categoryOf` 映射為正規類別。**無「狀態」欄**（狀態一律自動判定，見 3.4）。
  - **招生追蹤設定**（一次設定、帶入期數成效管理）：類別＝個案共學會時顯示「說明會場次」數量設定（預設 4，1–8 可增減，建議 3–5）；其他類別不顯示。**已移除「招生對象·來源標籤」欄**（原本的 tag 不真正帶人進 pool、會誤導）；招生名單一律在「期數成效 › 當期報名追蹤」匯入。`course.seminarCount` 仍存；`recruitSources` 保留欄位但新建預設空。
  - 複製模式：自動帶入不重複新代碼（來源-2、-3…）、名稱加「（複本）」。
  - 編輯模式：載入資料並就地儲存（`儲存變更`）；精靈內不編輯訂單，但**原樣保留 `initial.orders` / `initial.revenue` / `initial.students`（不可清空）**。
- **② 課程場次**（`SessionPlanEditor`）：新增/刪除場次，每場填日期（自動顯示星期）、時間；每場底下可新增**無上限**個案（主題/情境副標/講師/服務單位），個案可移除；底部統計「N 場 · M 個個案」。**講師欄串師資檔案**（見 3.9）：輸入可搜尋 `FACULTY`，選定後自動帶入服務單位，也可手打新講師。
- **②（末步）課程場次後即可建立**；本階段精靈不設成本（已移除原「成本結構」步驟）。新建 `costs={}`/`revenue=0`（毛利＝營收−成本＝0）；編輯原樣保留 `initial.costs`/`initial.revenue`。`course-finance.jsx` 與課程詳情頁「損益表」匯出不動；營收仍來自課程管理的訂單匯入（見 3.3a）。

### 3.3 課程詳情頁（`CourseDetailView`）
- 麵包屑：課程管理 ＞ 課程名稱。
- 摘要 KPI：學員人數、場次數、補課截止、主講、狀態。停課時顯示紅色提示橫條。
- Topbar 動作：**新增單堂課程／新增學員／停課・復課／損益表（匯出 .xls）／前往點名**。課程詳情頁新增「新增學員（後台訂單匯入）」：`AddStudentsModal` 沿用 `StudentOrdersEditor` 系統匯入（`importOnly`）讀 `BACKEND_ORDERS`，匯入寫 `course.orders` 並同步 `students`/`revenue`（`updateCourse` 寫透 cache）；同商品依 `src` 去重。**現階段唯一學員來源**，手動名單增減留在出缺勤頁（見 3.8）。
- 場次表：堂次／上課日期（含星期時間）／主題（每個案主題＋情境副標）／主講（各個案講師）／出席率／狀態／點名鈕。
- **左鍵點整列 → 進入該堂點名**；**右鍵某一堂**選單：**編輯此堂內容／前往點名／刪除此堂**（`SingleSessionEditModal`；刪除後自動重編堂次）。
- 空狀態（無場次）：提供「編輯場次與個案」按鈕。

### 3.3a 新增學員（後台訂單匯入，`AddStudentsModal`）★ 唯一學員來源
- 課程詳情頁 Topbar「新增學員」開 Modal，沿用 `StudentOrdersEditor` 的**系統匯入**（`importOnly`，隱藏手動建立）：讀 `BACKEND_ORDERS`，選後台商品（＝課程）帶入全部訂購人與金額；一筆訂單＝一位學員。
- 匯入寫 `course.orders` 並經 `updateCourse` 持久化（寫透 cache），同步 `course.students`（筆數）與 `course.revenue`（金額加總）。如此損益表、場次出席率、期數成效 `recruitingPool` 與名單 `enrolledFor` 自動沿用同一份資料。
- **去重**：同一商品避免重複匯入（`imported` set 依現有 orders 的 `src` 推導，已匯入者顯示「已匯入」並 disabled）。
- **不做**手動新增/刪除/修改學員；個別調整留待下階段與出缺勤頁名單編輯整合為單一入口（見 3.8）。

### 3.4 課程狀態（`deriveCourseStatus`）
- 四種：**招生中／上課中／已結案／停課**。
- **建課時填兩段日期**：招生期間（`recruitStart`／`recruitEnd`，選填，供顯示與招生追蹤）與開課期間（`startDate`／`endDate`，必填）。狀態由系統依今日自動判定：今天 < 開課日＝招生中；開課日 ≤ 今天 ≤ 結課日＝上課中；今天 > 結課日＝已結案。**無手動狀態選單**；需延長/調整改於「編輯課程」修改日期。
- **停課**為唯一手動狀態，於 Topbar「停課／復課」鈕設定，持久化（`hbrCRM.suspendedCourses`），**優先於日期規則**。

### 3.5 成本結構與毛利（`course-finance.jsx`）
- **11 項固定成本**（單一金額欄，目前標籤）：場地租賃費、講師費、硬體設備租賃用、軟體與線上平台費、場地布置費、固定行銷與廣告宣傳費、工作人員費、攝錄影費、交通住宿費、活動保險費、金流手續費。
- **雜支**：分項清單 `miscItems:[{name,amount}]`，可多筆（費用名稱＋金額），便於統計。
- **營收**：由學員訂單金額加總得出，**不可手動輸入**；新建課程＝0。學員訂單於課程詳情頁「新增學員」匯入（見 3.3a），以 `course.orders` 加總覆寫 `revenue`；編輯課程精靈不動訂單，原樣保留 `initial.revenue`。
- 毛利 = 營收 −（固定成本合計 + 雜支小計）；毛利率 = 毛利 / 營收。即時計算（`calcFinance`）。

### 3.6 匯出 / 上課名單管理
- **學員資料來源：課程詳情頁「新增學員」（後台訂單匯入 → `course.orders`，見 3.3）為唯一入口**。「實際上課人員」的個別增減（手動名單編輯）一律在出缺勤（點名）頁，不在課程管理另做一套，以免資料源打架。
- **上課名單的「手動增/減/修改」一律在出缺勤（點名）頁，不在課程詳情（單堂列表）頁**。進入路徑：課程管理 → 選期數 → 單堂列表 → 點選某堂 → 出缺勤頁「單場次點名」「編輯名單」。進入路徑：課程管理 → 選期數 → 單堂列表 → 點選某堂 → 出缺勤頁「單場次點名」「編輯名單」。
- 「編輯名單」開 `RosterEditModal`：**預設帶出購買該期的所有訂單會員（`customersForCourse`）＋自動匯入＋手動增刪＋儲存（回填該期預設名單）＋匯出 .xls**；「實際上課人員不一定」故可在此調整。
- **名單資料源一律走 `window.DATA.loadRoster(courseId)`**（自動推導報名學員＋既有 override），**不可**直接讀種子 `ROSTER[id]`（種子僅 ICL-16 有，其餘期會誤判為空）。
- 課程詳情頁仍以 `loadRoster` 取名單，但**僅用於場次出席率計算**，不提供名單編輯入口。
- **學員名單 .xls**：該課程所有學員（學員編號/姓名/公司/職稱）。
- **損益表 .xls**：營收、11 項固定成本、雜支（小計＋分項明細）、成本合計、毛利、毛利率、各項佔營收比。

### 3.7 課程設定（`CourseSettingsModal`）

- **場次內容**：「編輯單堂內容」→ 開 `SessionEditModal`，可後續調整所有場次的日期/時間/個案。

### 3.8 上課名單（Roster）★ 重要契約
- 名單是**課程層級**，套用所有場次；只能透過 `loadRoster(courseId)` / `saveRoster(courseId, list)` 存取。
- 預設＝**自動匯入開期所有上課學員**：`defaultRoster()` → 優先種子 `ROSTER`，否則 `enrolledFor()`——合併「客戶名單中購買此課程者 `customersForCourse()`」與「課程自身訂單 `course.orders`（精靈步驟③ 系統匯入／手動建立）」，依 id/姓名去重。故新建課程在精靈匯入的學員，進點名頁即為預設名單。
- **Given** 使用者在點名頁「編輯名單」**手動新增**學員 **When** 儲存 **Then** 寫入該期完整名單，**回填成該期預設名單**，之後所有場次與重新載入都讀同一份。不可存成場次層級。
- override 一旦存在即為權威；新報名需由「編輯名單 → 自動匯入」合併（依 id/姓名去重）。

### 3.9 師資檔案（Faculty）
- 講師主檔 `FACULTY[{id,name,org,cat}]`（`app/data.js`）；個案共學會 24 位引導師（來源：academy.hbrtaiwan.com/professor），`cat='個案共學會'`。
- 課程個案的「講師」欄為串檔輸入（`FacultyInput` + `<datalist>`）：**依課程類別以 `facultyFor(category)` 提供建議名單**（個案共學會→上述 24 位；其他類別尚無專屬師資時回退全部），可搜尋，**選定後自動帶入服務單位/職稱**；查無則以手打名稱沿用。
- 資料鍵沿用 `facilitator`（個案物件內），UI 一律顯示「講師」。
- 正式版：應改為 `faculty_id` 外鍵關聯師資主檔；目前以姓名比對為 demo 做法。

## 3b. 期數成效管理 規格（`performance.jsx`）★ 獨立模組

獨立於課程管理的 navigation 項目（課務區）。**三層結構**；課程管理本身維持原樣（只負責該課程的人員上課/點名）。
- **第一層 商品（＝課程系列／類別）列表**（`PerfProductList`）：依 `categoryOf` 將 `loadCourses()` 分組（個案共學會／實戰班／領導者俱樂部／其他），卡片顯示期別數與累計學員；個案共學會註明「含說明會成效追蹤」。

- **第三層 期 詳情（兩 tab，`PerfTermDetail`）**：
  - **① 當期報名追蹤表**（`EnrollTrackTable`）★ **匯入驅動**：招生對象池一人一列，**課程剛建立為空白**，不再掃整個客戶/潛客 DB。資料源 `recruitingPool(courseId)` 僅讀 `perf.poolMembers`（持久化於該 courseId）；成員記 refType(customer/prospect/external)、refId、姓名/公司/職稱/Email/電話、klass、sources。工具列三個匯入入口（皆開 Modal）：
    - **從潛客匯入**：列 `loadProspects()`，搜尋＋來源/狀態篩選，勾選加入 klass=舊潛客（帶 sources）；開關「標記為本期新潛客」勾了則 klass=新潛客並寫 `newProspectIds`。

    - **匯入外部名單**：手動多列（姓名/公司/職稱/Email/電話/來源複選），加入 klass=新潛客並寫 `newProspectIds`；同時一律在潛客管理建立潛客（`createProspectIfAbsent`，依 email/姓名跳過已存在者）。
    - **去重**：同一人（依 refId/姓名）池中僅一筆，重複自動略過並提示（`addPoolMembers`）。成員可從本期移除（`removePoolMember`，僅移出本期池，不刪底層）。
    - 報名/成交狀態每列就地編輯（存 `perf.track[key]`，key＝`C:`客戶id／`P:`潛客id／`E:`外部）。欄位：姓名、公司/職稱、身分分類、潛客來源、聯絡方式、**說明會（記到場次：未參加／第N場，選項依 `perf.seminars` 動態產生；每人實務上只參加一場）**、報名狀態、成交狀態、**當期備註**（每人×期獨立備註，存 `perf.track[key].note`）、移除。匯入 Modal（潛客/舊客）表頭有**全選**。**身分分類分頁**（仿 Topbar tabs）＋來源下拉＋搜尋；KPI 隨匯入動態更新。空狀態（池為空）置中提示＋匯入按鈕引導（池有名單時匯入鈕僅在右上工具列）。
  - **② 說明會成效追蹤表**（`SeminarPerfTable`）：**僅 `categoryOf===個案共學會` 顯示此 tab**。每期 3–5 場，**每場獨立計算、不跨場合併、不做歸因**。每場一列可編輯：發放參加通知數、回覆出席數、實際出席數、成交數；自動算 出席率＝實際出席÷回覆出席、轉換率＝成交÷實際出席。頂部 KPI ＋表尾「本期彙總」供橫向比較。資料走 `perf.seminars`（無則依 `course.seminarCount` 生空場）。**與追蹤表交叉對照**：每場場次卡下顯示「池內標記 M 人」（追蹤表該場標記數），KPI 加「池內標記出席 N」；若 N 大於實際出席總數顯示警示橫條提醒核對。手填數字保留（出席者可含未入池者），兩邊互不覆寫。
- 點側欄項目重新進入會重置回第一層（`key={navTick}` 強制 remount）；層間以 Topbar 麵包屑返回。
- 資料一律走 `app/data.js`（`loadPerf/savePerf/recruitingPool/addPoolMembers/removePoolMember`，仿 `loadProspects` 記憶體快取；鍵 `hbr_perf_v2`）。

### 3c. 潛客 → 顧客 轉換規則（訂單＋帳號比對）

- 觸發點：課程管理「新增學員（後台訂單匯入）」匯入訂單時，`convertProspectsByOrders(courseId, orders)` 逐筆比對潛客；比中者→轉為顧客（自潛客管理移除、於顧客管理建立/更新，purchased 加入本期課名），並於本期池 `track` 將該員 deal 標為「已成交」。

## 3d. 登入 / 帳號權限 規格（`auth.jsx` / `accounts.jsx`）★ 全 App 前置

- **登入閘門**：未登入時整個 App 只渲染 `LoginScreen`；登入成功後才進主介面（預設落地：課程管理）。`main.jsx` 以 `DATA.getSession()` 初始化 `user` state。
- **角色** `DATA.ROLES`：`superuser`（系統管理員，全權、唯一可進帳號管理）／`staff`（課務人員，日常操作）。其餘模組兩種角色都能用（本階段只管控帳號管理頁）。
- **登入**（`auth.jsx`）：Email＋密碼＋「記住我」。mock 驗證 `DATA.authenticate(email,password)` 回 `{ok,user}`／`{ok:false,reason}`（notfound/disabled/badpass，對應錯誤訊息）。記住我→`localStorage`、否則 `sessionStorage`（`setSession`）。登入頁附 demo 帳號提示（一鍵帶入）。
- **登出**：側欄左下角使用者頭像→選單「登出」→`clearSession()`＋回登入頁。
- **帳號權限管理**（`accounts.jsx`，**僅 superuser**）：列表（使用者/Email/角色/狀態/建立日期）＋角色分頁＋搜尋。操作：新增／編輯／停用·啟用／重設密碼／刪除。
- **安全規則**（superuser 限制設想）：
  - 至少保留一個「啟用中的 superuser」：不可停用／降權／刪除最後一個（`isLastActiveSuperuser`）。
  - 不可停用／刪除自己的帳號。
  - 新建帳號預設角色＝`staff`（課務人員），避免誤給高權。
  - 非 superuser **連側欄「帳號權限管理」項都不顯示**（`NAV` 項 `super:true`＋Sidebar 過濾）；即使手動導到 `accounts` 也會被導回課程管理（`main.jsx` 雙重防護）。
- 種子帳號：`admin@hbr.tw`/`admin123`（superuser）、`chen@hbr.tw`/`staff123`（staff，啟用）、`lin@hbr.tw`（staff，停用示範）。
- 正式版：改後端帳號系統與真正的密碼雜湊／權限驗證；前端只保留 UI 與 session 顯示。

## 4. 客戶管理 規格（`customers.jsx`）

- 勾選後底部中央浮出深色膠囊列：**批次修改／匯出所選／清除選取**。

### 4.4b 備註欄位 `note` ＋ 標籤 `noteTags`（自由編輯）
- 客戶物件含 `note`（自由文字）與 `noteTags`（標籤陣列）；皆供內部參考，不參與業務計算。

### 4.5 匯出（`customer-export.jsx`）
- 進入點：工具列「匯出」（範圍＝目前篩選結果／全部）、勾選後「匯出所選」、每列「匯出」、詳細抽屜「匯出」。
